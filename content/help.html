What to Do When You're Stuck

This course doesn't have answers included, and that's on purpose.

It's too easy to look at an answer, think "oh that makes sense" and move on without truly understanding it.

Also answers can rob you of the most important skills in coding: getting unstuck, and gaining confidence in your ability to find your own answers.

This section will give you some strategies to get unstuck.

First check the console

If something isn't working properly, first check the browser console.

If you have a syntax error (like you put a bracket in the wrong place or something), the console will show you which line number your error is on.

If you try to use a variable that doesn't exist yet, the console will catch that too.

The console is one of the quickest and easiest ways to catch errors, so start there.

Check your spelling and capitalization

Let's say you're trying to change the HTML inside a div.

If you use innerHTML, it will work properly because it's correct.

However, if you make a typo like "innerLMHT", the console won't show an error!

Why?

It's because you can store custom properties if you want to, so the browser just thinks you intended to store data in a custom property you've named "innerLMHT".

The same problem would happen if you typed "innerhtml". JavaScript is case-sensitive, so "innerhtml" is different from innerHTML.

If something's not working the way you expect, check your spelling and capitalization.

Test small pieces

If your code isn't working, break it down into the smallest possible pieces and test those.

For example if you're making a countdown clock, don't test the whole clock immediately. Instead start with the basics. Show a single number on the screen and see if that works. Then move onto the next step.

The smaller the pieces you test, the easier it will be to see where things went wrong.

Write what your code is doing

In order to get clarity on what your code is doing, break it down into small pieces and write in plain language what it is doing. I like to use comments for this purpose.

For example, let's say you have a loop that's showing a list of social media posts. Instead of saying "It's showing a list of posts," actually go into detail and say what's happening in each step of the loop.

A better example would be, "The first time the loop runs, i=0 and we're showing the first post (posts[0]). After this, i increases by 1, and we go through the cycle again. This repeats as long as i is less than the total number of posts." And just continue explaining everything like this.

Often, this technique leads to a moment where you say something like "oh, the variable i is not what I expected it to be here" or "oh, that loop ends too early" or "hmm I was getting the number of the post, but I never actually used that number to get the post itself!"

When in doubt, write it out.

Start over and write what you want your code to do

This is similar to the previous technique except instead of writing what your existing code is doing, you start over again and write what you want your code to do.

Again, the smaller the pieces, the better.

The benefit of starting over is that you won't be distracted by previous code, and it can be much easier to think with a clean slate.

When you're doing a complex project, you don't want to do your thinking and your coding at the same time. Write in plain language first to clarify your thinking and avoid being distracted by syntax concerns.

Look back through the lessons

Self-explanatory.

If you're stuck look back through the lessons and you might find a hint.

If you find there's something from a previous lesson you don't quite remember, play with it until you get comfortable again.

Google your specific issues

If you've done the previous steps up to now and you're still stuck, then you have a great foundation to start Googling things.

By breaking down your problems into small parts, you can look up specific things instead of floundering for general answers.

Before Googling something, narrow down the problem as much as possible.
